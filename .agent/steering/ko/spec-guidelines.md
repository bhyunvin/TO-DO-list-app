# Spec 문서 작성 가이드라인

## 언어 요구사항

### 주 언어: 영어
모든 spec 문서(requirements.md, design.md, tasks.md)는 주 언어로 영어로 작성되어야 합니다.

**근거:**
- 영어는 기술 문서의 보편적인 언어입니다
- 국제 개발자와의 협업을 가능하게 합니다
- 코드 주석 및 변수명과의 일관성을 유지합니다
- 개발 도구 및 AI 어시스턴트와의 통합을 용이하게 합니다

### 한글 번역: 선택사항
세 개의 spec 문서를 영어로 완성한 후, 한글 번역을 생성할 수 있습니다:

1. spec 디렉토리 내에 `ko/` 하위 디렉토리 생성
2. 세 개의 문서(requirements.md, design.md, tasks.md) 모두 번역
3. 번역된 파일을 `ko/` 하위 디렉토리에 배치

**예시 구조:**
```
.kiro/specs/feature-name/
├── requirements.md          # 영어 (필수)
├── design.md               # 영어 (필수)
├── tasks.md                # 영어 (필수)
└── ko/                     # 한글 번역 (선택사항)
    ├── requirements.md
    ├── design.md
    └── tasks.md
```

## 작성 스타일

### 요구사항 문서
- EARS (Easy Approach to Requirements Syntax) 패턴 사용
- INCOSE 의미론적 품질 규칙 준수
- 사용자 스토리를 영어로 작성
- 명확하고 모호하지 않은 언어 사용

### 설계 문서
- 기술적인 영어 용어 사용
- 영어 주석이 포함된 코드 예제 포함
- 전문적인 기술 작성 스타일 유지
- 적절한 경우 Mermaid 다이어그램 사용

### 작업 문서
- 작업 설명을 영어로 작성
- 동작 동사 사용 (Implement, Create, Refactor, Test 등)
- 요구사항 ID를 사용하여 요구사항 참조
- 작업 설명을 간결하고 실행 가능하게 유지

## 코드 주석 vs. Spec 문서

**중요한 구분:**
- **코드 주석**: 한글로 작성 (프로젝트 표준에 따라)
- **Spec 문서**: 영어로 작성 (이 가이드라인에 따라)

이러한 분리는 다음을 보장합니다:
- Spec은 국제 협력자가 접근 가능
- 코드는 한국 개발팀을 위해 현지화됨
- 계획(spec)과 구현(코드) 간의 명확한 구분
